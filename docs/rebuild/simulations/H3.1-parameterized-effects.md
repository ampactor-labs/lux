# Handle 3.1 — Parameterized Effects

*Role-play as Mentl, tracing what happens when Morgan writes
`with Sample(44100)` in the Sonic Pulse pipeline. Names the layers
where the substrate currently treats every effect as a flat string,
and resolves what changes so the same effect at different parameter
values is a different row.*

---

## The scenario

Three real-time-audio fragments from `examples/sonic_pulse.ka`:

```
fn smooth(stream) with Sample(44100) =
  stream <~ delay(3)

fn pulse(audio, ctrl) with Sample(44100) + IO + Network + !Alloc =
      (audio |> analyze)
    ><
      (ctrl  |> smooth)
    |> merge
    ~> sink_ws("wss://pulse.local/feed")
    ~> clock.tick(44100)
```

`Sample(44100)` is a row entry — not a function call, not a type
parameter. Two installations of `Sample` at different rates
(`Sample(44100)` vs `Sample(48000)`) must be DISTINCT in the algebra:
unification fails, subsumption fails, an `audio_at_44k` function does
not satisfy a `with Sample(48000)` constraint.

What has to work? Trace each layer.

---

## Layer 1 — Parser (today: name-only)

`parse_one_effect` (parser.ka) returns `(name, negated)`. The
`with` clause consumes a `+`-separated list of effect tokens; each
token is `[!] Identifier`.

The Sonic Pulse line `with Sample(44100) + IO + Network + !Alloc`
needs four effects:
- `Sample(44100)` — parameterized, positive
- `IO` — bare, positive
- `Network` — bare, positive
- `!Alloc` — bare, negated

Parser change: after consuming the identifier, peek for `LParen`. If
present, consume an argument list (literal-only for v1) and emit
the parameterized form. Otherwise the bare form.

Argument form (v1): scalar literals — `TInt(N)`, `TFloat(F)`,
`TString(S)`. Identifier arguments (e.g., `Clock(monotonic)`),
expressions, and named constants are deferred (separate walkthrough).

Returned shape (v1): `(name, args, negated)` where `args` is a
`List<EffArg>`. Today's `(name, negated)` callers thread the
extra slot.

Verdict: **mechanical parser change**. One new ADT (EffArg), one new
helper (parse_eff_args), and the call shape grows by one slot.

---

## Layer 2 — Types / Row algebra (today: List<String>)

### The current substrate

`EfClosed(List)` carries effect names as **strings**. Set operations
(`set_union`, `set_diff`, `set_contains`) compare names with
`str_eq`. This works precisely BECAUSE names are flat.

The moment row entries carry arguments, string equality is wrong:
`Sample(44100)` ≠ `Sample(48000)` even though their NAMES match.

### The shape that matches the algebra

```
type EffArg
  = EAInt(Int)
  | EAFloat(Float)
  | EAString(String)

type EffName
  = ENamed(String)                 // bare effect — IO, Alloc, ...
  | EParameterized(String, List)   // name + List<EffArg>
```

`EfClosed` rewrites: `EfClosed(List<EffName>)`. `EfOpen` similarly.

### Equality, ordering, set operations

**EffName equality** (eff_name_eq): names match AND (for parameterized)
arg lists pointwise equal.

**EffName ordering** (eff_name_lt): primary key is name; secondary key
is arg list lex order. Ordering matters because `EfClosed` is sorted
and deduped; the substrate-wide invariant is that two rows with the
same logical content have identical representations.

**Set operations**: rewrite `set_union` / `set_diff` / `set_contains`
to take the comparator. Either pass the comparator as a parameter
(verbose every site) or split the row-level set algebra into its own
module that knows EffName specifically. The latter is cleaner — the
runtime/strings.ka set operations stay string-typed; effects.ka gets
its own row-set surface (`name_set_union`, etc.).

### Show / display

`show_effrow` writes `Sample(44100)` for the parameterized form. Args
render as comma-separated literals. Diagnostic copy stays readable.

Verdict: **substrate sweep**. Every site that constructs, compares,
or formats a row name updates from `String` to `EffName`. Bounded:
~12 sites in effects.ka, ~6 in infer.ka, a few in mentl.ka.

## Layer 3 — Effect-op registration (today: implicit param-free)

`register_one_op` (infer.ka) builds:

```
TFun(params, ret, mk_ef_closed([effect_name]))
```

Today `effect_name` is a String. After H3.1: an `EffName`. Bare
effect declarations (the only kind v1 supports at the DECLARATION
LEVEL) bind `ENamed(name)`. The op's row uses the bare form because
parameter binding happens at INSTALL time, not at declaration.

EffectOpScheme stays `EffectOpScheme(String)` — the declaring
effect's name as a string is the right metadata; SchemeKind doesn't
need to know the install args (those vary per call site).

Verdict: **one-line type change** at the registration site. The op's
type is parametric over the effect's args; the args appear in the
row only at install.

---

## Layer 4 — `with` install (today: name-list-to-strings)

`with Name1 + Name2 + !Name3` parses today as a list of
`(name, negated)` tuples. Inference's `build_declared_row` folds
them into an `EfRow`. `mk_ef_closed`, `mk_ef_neg`, intersection.

After H3.1: each entry is `(EffName, negated)`. `build_declared_row`
threads the EffName through. `mk_ef_closed([eff_name])` constructs
the singleton row.

Subsumption (`row_subsumes`): for each name in the smaller row,
check `name_set_contains` of the larger. The contains check now
compares EffName with eff_name_eq, not str_eq.

Verdict: **mechanical**. The shape change at the row-name level
propagates through the existing algebra without restructuring it.

---

## Layer 5 — Lowering / Emit (today: row-blind)

Lower doesn't read effect names — it reads row GROUNDNESS
(`row_is_ground`) for monomorphic detection. Args don't change
groundness. No lower change.

Emit (`backends/wasm.ka`) doesn't read effect names either — handler
arms are bound by op_name, not by effect_name + args. The emit-time
identity of `Sample(44100)` vs `Sample(48000)` is irrelevant — both
install handlers whose ARM dispatch uses `$op_<op_name>`.

Verdict: **no change**. Parameterized effects are a TYPE-LEVEL
construct; runtime is unchanged.

---

## Layer 6 — Diagnostics (today: name-only error copy)

`E_EffectMismatch` today says: "function declares X but body
performs Y". With parameterized effects, X and Y can be:
- `Sample(44100)` vs `Sample(48000)` — parameter mismatch
- `Sample(44100)` vs `Sample` — bare/parameterized mismatch

Both surface naturally through `show_effrow` once it knows the
parameterized form.

Mentl's audit (H5) gains a candidate: when a function declares
`Sample(44100)` and a callee declares `Sample`, suggest tightening
the callee. Out of scope for H3.1 — the substrate makes the
suggestion possible; the surface lands with H5.

---

## Layer 7 — Negation (today: `!Name` is a flat string)

`!Sample(44100)` means "any row entry except Sample at exactly rate
44100". `EfNeg(EfClosed([ENamed("Sample", [EAInt(44100)])]))`
normalizes the same as today's negation, just with the parameterized
EffName inside.

`!Sample` (bare negation of a parameterized effect) means "any row
entry except any Sample, regardless of args". This is a SEPARATE
question from H3.1 — the v1 design says: bare-name negation matches
ONLY the bare form, parameterized negation matches its specific
parameterization. Wildcarding across args is a v2 question.

Verdict: **no algebra change**. Negation works through the existing
sub/inter machinery; the only delta is the EffName payload.

## Design synthesis (for approval)

**EffName ADT.** New ADT in types.ka:

```
type EffName
  = ENamed(String)
  | EParameterized(String, List)

type EffArg
  = EAInt(Int)
  | EAFloat(Float)
  | EAString(String)
```

**EfClosed / EfOpen rewrite.** Carry `List<EffName>` instead of
`List<String>`. The substrate-wide sorted-deduped invariant holds via
new comparators.

**Row-level set algebra.** New name_set_* operations in effects.ka:
`name_set_union`, `name_set_diff`, `name_set_contains`,
`name_set_eq`. Mirror the string-set surface in runtime/strings.ka
but compare EffName.

**Parser change.** `parse_one_effect` returns `(EffName, negated)`.
After identifier, peek for LParen; consume `parse_eff_args` if
present.

**Row construction.** `mk_ef_closed`, `mk_ef_open`, `mk_ef_neg` take
List<EffName>. `build_declared_row` threads through.

**Show.** `show_effrow` formats `Sample(44100)` correctly.

**No lower / emit change.** The construct is type-level only.

## What H3.1 reveals (expected surprise)

- **String-keyed maps over effect names need to migrate too.**
  effect_name_to_op_table maps String→ops. If we ever key by
  EffName, the comparator changes. Today no such map exists; if
  H3.1 prompts one, it should be EffName-keyed from day one.

- **Show / parse round-trip is the canonical equality witness.**
  `Sample(44100)` parses to `EParameterized("Sample", [EAInt(44100)])`;
  `show_effrow` writes back `Sample(44100)`. Round-trip equality is
  the substrate-level test.

- **EffectOpScheme stays String.** The op's declaring effect is
  identified by NAME — args belong to the install. SchemeKind doesn't
  need to know install args; they vary per call site. This is a
  separation of concerns: declaration metadata (SchemeKind) vs
  install metadata (row payload).

- **No runtime cost.** The args live in the type system. WAT emission
  doesn't see them. The row algebra distinguishes Sample(44100) from
  Sample(48000) at compile time; the wasm has no notion of either.

- **Bare/parameterized mismatch is its own diagnostic class.** If a
  body performs an op declared by `effect Sample` but the function
  declares `with Sample(44100)`, that's a mismatch — but a different
  kind than rate inequality. The diagnostic copy should distinguish
  ("Sample requires a rate; got bare Sample" vs "Sample(44100)
  expected, Sample(48000) found").

## Ordering in the cascade

H3.1 lands AFTER H3 because:
- ConstructorScheme machinery from H3 is a peer mechanism
  (registration metadata that downstream layers dispatch on)
- The shape parallels: SchemeKind dispatch in lower mirrors EffName
  dispatch in row algebra
- One walkthrough per cascade step keeps reasoning loadable

H3.1 lands BEFORE H2 because Sonic Pulse uses both — and the
showcase-by-tracing (post-cascade) cannot trace until both work.
