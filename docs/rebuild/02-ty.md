# 02 — Ty: the type ADT, refined and ownership-aware

**Purpose.** Lock the Ty ADT as ground truth for every downstream
spec. Covers refinement types, continuation types, `TParam` wrapping
for ownership markers, and `TVar` as a handle into the SubstGraph.

**Kernel primitives implemented:** #1 (SubstGraph handles as `TVar`),
#5 (`TParam` carries `Own | Ref | Inferred`), #6 (`TRefined(Ty, Predicate)`),
and the substrate on which #2's MultiShot resume discipline operates
(`TCont(ret, OneShot | MultiShot | Either)`).

- Generic Refinement Types POPL 2025 — per-call-site refinement
  instantiation via unification.
- Affect POPL 2025 — one-shot / multi-shot resume distinction at the

## ADT

```lux
type Ty
  = TInt
  | TFloat
  | TString
  | TUnit
  | TList(Ty)
  | TTuple(List)
  | TFun(List, Ty, EffRow)      // params (TParam list), return, effects
  | TVar(Int)                   // handle into SubstGraph (spec 00)
  | TName(String, List)         // nominal type + type args
  | TRecord(List)               // fields: [(name, Ty)]
  | TRecordOpen(List, Int)      // fields + row-variable handle
  | TRefined(Ty, Predicate)     // NEW — refinement type
  | TCont(Ty, ResumeDiscipline) // NEW — handler continuation type

type TParam
  = TParam(String, Ty, Ownership)    // NEW — wraps a parameter

type Ownership
  = Inferred                    // compiler decides (default)
  | Own                         // affine: consumed at use
  | Ref                         // borrowed: cannot escape

type ResumeDiscipline
  = OneShot
  | MultiShot
  | Either
```

Refinement predicates (structural; solver is a handler in Arc F.1):

```lux
type Predicate
  = PTrue
  | PEq(PExpr, PExpr)
  | PLt(PExpr, PExpr)
  | PLeq(PExpr, PExpr)
  | PAnd(Predicate, Predicate)
  | POr(Predicate, Predicate)
  | PNot(Predicate)
  | PCall(String, List)         // user-defined; solver handles

type PExpr
  = PSelf                       // the value being refined
  | PInt(Int)
  | PFloat(Float)
  | PString(String)
  | PVar(String)
  | PApp(String, List)
```

## TVar is a handle, not a binding

`TVar(42)` does NOT carry the resolved type inline. It is an index
into the SubstGraph (spec 00). To resolve:

```lux
perform graph_chase(42)    // returns GNode(NBound(t), r) — or the handle is NFree
```

Observers (show_type, lookup_ty in lowering, query) all chase.
There is no `apply(subst, ty)` function; the graph chase replaces
any subst-apply pattern. No separate substitution table — the graph
IS the substitution.

**Show-type produces two shapes:**
- Resolved: structural walk, chasing TVars as it encounters them.
- UNRESOLVED: if chase returns `NFree(e)`, printer emits
  `TVar(42)@epoch=e`. Signal to the query layer that inference
  hasn't ground this yet.

## Refinement: structure in Phase 1, verification via effect

`TRefined(TInt, PAnd(PLeq(PInt(1), PSelf), PLeq(PSelf, PInt(65535))))`
is well-formed from day one. The ADT, predicates, and unification
rules ship from day one. **Verification is mediated by the `Verify`
effect (spec 06), not by a stub.**

Unification rules — each refinement unification performs `Verify`,
which accrues a structural proof obligation rather than silently
accepting or rejecting:

- `TRefined(a, p) ~ TRefined(b, q)` → unify `a ~ b`; perform
  `verify(span, PAnd(p, q), Refinement(p, q))`.
- `TRefined(a, p) ~ b` → unify `a ~ b`; perform
  `verify(span, p, Refinement(p, PTrue))`.
- `a ~ TRefined(b, q)` → unify `a ~ b`; perform
  `verify(span, q, Refinement(PTrue, q))`.

**Handler tiers (spec 06):**
- **Default: `verify_ledger`** — records every obligation
  into a structured debt record (span, predicate, reason). At compile
  end, prints `N verification obligations pending`. Code compiles;
  obligations are visible, countable, and queryable via
  `inka query "verification debt"`.
- **Arc F.1: `verify_smt`** — dispatches predicates to Z3 (nonlinear
  arithmetic), cvc5 (finite-set/bag/map), Bitwuzla (bitvectors) by
  residual form. Handler swap; source code unchanged.

No stub. No silent acceptance. No accumulating warnings. The gap
between "no solver" and "full solver" is hosted by the handler
mechanism itself — the first structural demonstration of the
handler-swap thesis in the compiler's own core.

## Ownership: structure in Phase 1, runtime in Arc F.4

`TParam(name, ty, Own|Ref|Inferred)` is the compile-time interface to
the `Consume` effect (spec 07). The Ty ADT carries the marker so
effect inference can see it; allocation and affine-tracking logic
live in the handler, not in the type.

TFun equality includes ownership: `fn(Own(Int)) -> Int` is NOT equal
to `fn(Ref(Int)) -> Int`. Subsumption:
- `Own ⊆ Inferred` (calling with an inferred slot accepts an Own
  value).
- `Ref ⊆ Inferred` (same for Ref).
- `Own ≠ Ref` (incompatible disciplines).

## TCont: continuation types

Plain function types are `TFun`. Continuation types captured inside
handler bodies are `TCont(ret, discipline)`. The distinction carries
through inference so scoped-arena × multi-shot (Arc F.4) has the type-
level lever Affect POPL 2025 documents.

`TCont` is a distinct variant. The handler classification
(TailResumptive / Linear / MultiShot, spec 05) distinguishes at the
IR level; `TCont` lifts that discrimination into the type layer where
other passes can see it.

## Env and Scheme

The Env ADT carries `Scheme`-valued bindings for let-generalization:

- `type Scheme = Forall(List[Int], Ty)` — a monomorphic binding is
  `Forall([], ty)`.
- `env_extend(env, name, scheme, reason) -> Env`.
- `env_lookup(env, name) -> Option((Scheme, Reason))` — `None` when
  unbound; callers destructure
  `match env_lookup(env, n) { None => ..., Some((sch, r)) => ... }`.
- `env_with_primitives` installs Int, String, Bool, List, Option at
  compilation start.

No `apply(subst, ty)` / `subst_empty` / `subst_set`. No separate
substitution table. `graph_chase` (spec 00) replaces the entire
subst-apply pattern; the graph IS the subst.

## Display

`show_type` and `show_ty` are the canonical printers. TVars resolve
by performing `graph_chase`, never by pattern-matching a subst
argument.

## Rejected alternatives

- **Inline ownership on TFun instead of wrapping TParam.** Bloats
  TFun's constructor. Separating as TParam keeps TFun shape stable
  and localizes ownership concerns.
- **Dependent types end-to-end (Idris 2, Lean 4).** Refinement types
  cover 80% with decidable checking and no user-visible proof terms.
  The ergonomic tax is the lesson.
- **Path-dependent TVar (TVar-per-scope).** Graph overlay (spec 00)
  gives module-scope; per-scope inside a module over-engineered.
- **ResumeDiscipline as TFun field.** Cluttering TFun for a case used
  only inside handlers. `TCont` keeps surfaces orthogonal.
