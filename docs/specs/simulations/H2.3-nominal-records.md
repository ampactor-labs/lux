# Handle 2.3 — Nominal Record Types

*Role-play as Mentl, tracing what happens when Morgan declares
`type Person = {name: String, age: Int}` and writes
`Person{name: "Morgan", age: 30}`. Names where structural records
(H2 v1) leave a gap and resolves how nominal records use the
SchemeKind machinery H3 already built.*

---

## The scenario

The Sonic Pulse showcase opens with:

```
type Audio    = {ts: Float, sample: Float}
type Spectral = {centroid: Float, flux: Float, dominance: Float}
```

And constructs:

```
fn analyze(frame: List<Float>) with !Alloc =
  frame
    |> spectral.fft
    <| (spectral.centroid, spectral.flux, spectral.dominance)
    |> fn (c, f, d) => Spectral{centroid: c, flux: f, dominance: d}
```

Two nominal record TYPES are declared. One nominal record is
CONSTRUCTED (`Spectral{...}`) at the lambda's tail. After H2 v1
the language can:
- declare `{name: Ty, age: Ty}` types inline
- construct `{name: ..., age: ...}` literals
- access `r.name` field
- pattern destructure `{name, age}`

But it CANNOT:
- declare `type Audio = {...}` and have `Audio` resolve to a record type
- construct `Audio{...}` with type-name prefix
- distinguish `Audio` from `Spectral` at the type level (both are
  structurally `{centroid: Float, flux: Float}` … no, wait — they're
  different shapes; but two records with the SAME shape would be
  structurally indistinguishable)

H2.3 closes the gap. After it lands, nominal records work and the
showcase's opening lines parse + check + lower correctly.

What has to work? Trace each layer.

---

## Layer 1 — Parser (today: half done)

### Type declaration with record body

`type Person = {name: String, age: Int}` is currently parsed as
TypeDefStmt. The parser already accepts `type Name = ...` and
forwards to parse_type_variants for ADT bodies. Need: detect when
`{` follows `=` in a type decl, dispatch to record-type body parsing.

```
fn parse_type_decl(tokens, pos) = {
  // ... existing: name, type-params, `=`
  let p = skip_ws(tokens, after_eq)
  match kind_at(tokens, p) {
    TLBrace => {
      // Record type declaration
      let (flds, p2) = parse_record_type_fields(tokens, skip_ws(tokens, p + 1))
      let body = TRecord(sort_record_type_fields(flds))
      // Wrap as a single-variant TypeDef with record body
      TypeDefStmt(name, type_params, [(name, [body])])
    },
    _ => parse_type_variants(...)   // existing ADT body
  }
}
```

`parse_record_type_fields` and `sort_record_type_fields` already
exist (H2). Reuse.

### Type-name-prefixed record literal

`Person{name: "Morgan", age: 30}` — parsed today as VarRef("Person")
followed by a brace. The brace is NOT part of any existing primary
expression, so parsing fails or splits. The fix: post-VarRef, peek
for `{` and dispatch to nominal-record-literal parsing.

The disambiguation is delicate — `f { ... }` could mean "function
call with block argument" in some languages, but Mentl uses `f(...)`
for calls and `{...}` for blocks/records. So `Person{...}` is
unambiguous IF the brace immediately follows the identifier (no
whitespace). To avoid surprising `Person {block}` interpretations:

**Decision (per SYNTAX.md spirit):** the brace MUST IMMEDIATELY
follow the type-name identifier (no whitespace). Otherwise it's
two separate expressions.

```
// In parse_postfix or parse_primary suffix loop:
match kind_at(tokens, pos) {
  TLBrace => {
    // Check if previous token was identifier with no whitespace gap
    if pos > 0 && is_capitalized_ident_at(tokens, pos - 1) &&
       no_ws_between(tokens, pos - 1, pos) {
      let type_name = ident_at(tokens, pos - 1)
      let (sorted, p) = parse_record_literal_body(tokens, pos + 1)
      (NamedRecordExpr(type_name, sorted), p)
    } else { ... }
  }
}
```

Or simpler — fold into the suffix-chain after parse_primary returns
a VarRef whose name is capitalized.

Either way: a new AST variant.

```
// Add to Expr:
| NamedRecordExpr(String, List)         // type_name, [(field, value)] sorted
```

Verdict: **moderate parser work**. New AST variant, new disambig
rule, reuse of H2's record-field machinery.

## Layer 2 — Inference

### TypeDefStmt with record body — registration

Today `register_type_constructors` walks variants assuming each is
ADT-style `(ctor_name, field_types)`. For a record type, the
"variant" IS the record body. Treat it as a single variant whose
constructor IS the type name and whose fields are the record fields:

```
register_type_constructors("Person", [], [("Person", [TRecord(fields)])])
```

This produces an env entry:
```
"Person" → Forall([], TFun([TParam("_0", TRecord(fields), Inferred)],
                            TName("Person", []),
                            EfPure))
            with kind ConstructorScheme(0, 1)
```

But that's WRONG for nominal records — `Person{name: ..., age: ...}`
isn't a function call with one TRecord argument; it's a CONSTRUCTION
with named fields. The right shape for the env entry is:

```
"Person" → Forall([], TName("Person", []))
            with kind RecordSchemeKind(fields)
```

`RecordSchemeKind(fields)` carries the declared field types so:
- NamedRecordExpr inference reads the schema, type-checks each
  literal field against the declared type.
- lower's NamedRecordExpr dispatches on RecordSchemeKind to lower
  as LMakeRecord (same as bare record literals — type-name distinction
  is type-system-only).

### SchemeKind extension

```
type SchemeKind
  = FnScheme
  | ConstructorScheme(Int, Int)
  | EffectOpScheme(String)
  | RecordSchemeKind(List)               // [(field_name, ty)] declared
```

Sites updated:
- `register_one_variant` — when the variant's body is a single
  TRecord, register as RecordSchemeKind instead of ConstructorScheme.
- lower's CallExpr / NamedRecordExpr dispatch — read the kind, route
  accordingly.

### NamedRecordExpr inference

```
NamedRecordExpr(type_name, fields) => {
  // Look up type_name's RecordSchemeKind
  match env_kind_of(type_name) {
    Some(RecordSchemeKind(declared_fields)) => {
      // For each provided field, find the declared type and unify.
      // Verify every declared field is provided (no missing).
      // Verify no extra fields (no surplus).
      check_nominal_record_fields(declared_fields, fields, span)
      // Bind handle to TName(type_name, [])
      perform graph_bind(handle, TName(type_name, []),
        Located(span, Inferred("nominal record literal")))
    },
    Some(_) => /* error: not a record type */,
    None    => /* error: undeclared type */
  }
}
```

`check_nominal_record_fields` walks both lists in sorted order
(both already sorted) and:
- on name mismatch → E_RecordFieldMismatch
- on missing declared field → E_RecordFieldMissing
- on extra field → E_RecordFieldExtra
- on type mismatch → unify and let the unification error carry

### Field access on nominal records (already works)

`p.name` where `p: Person`:
- infer's FieldExpr arm looks up `p`'s type (TName("Person", [])).
- resolve_field_offset needs to walk TName → look up Person's
  RecordSchemeKind → find field's offset.

This is a SMALL addition to resolve_field_offset (lower.mn):

```
fn resolve_field_offset(rec_handle, field_name) =
  match perform lookup_ty(rec_handle) {
    TRecord(fields)        => field_byte_offset(fields, ...),
    TRecordOpen(fields, _) => field_byte_offset(fields, ...),
    TName(type_name, _)    => match env_kind_of(type_name) {
      Some(RecordSchemeKind(fields)) => field_byte_offset(fields, ...),
      _ => 0
    },
    _ => 0
  }
```

### Pattern matching on nominal records

`match p { Person{name, age} => ... }` — wait, do we even support
type-name-prefixed patterns? Sonic Pulse doesn't use this form.
Reasonable to defer: structural patterns `{name, age}` work against
nominal records too because the pattern's TRecordOpen unifies with
TName(record_type) via lookup-through-SchemeKind. Type-prefixed
record patterns are sugar; defer.

## Layer 3 — Lowering

NamedRecordExpr lowers identically to MakeRecordExpr:

```
NamedRecordExpr(_type_name, fields) =>
  LMakeRecord(handle, lower_record_field_values(fields))
```

The type name doesn't survive to LIR — runtime sees raw field
storage. The type system has done its work; lower just emits the
record.

## Layer 4 — Emission

No change. LMakeRecord is unchanged. Field offsets via existing
emit_record_field_stores.

## What H2.3 closes

- `type Audio = {...}` declarations parse and register.
- `Audio{ts: 0.0, sample: 1.0}` constructs typed nominal records.
- `a.ts` resolves correctly through TName → RecordSchemeKind.
- Nominal type identity holds: `Audio` and `Spectral` with same
  field shapes are still distinct types (TName-keyed).
- Sonic Pulse's opening lines parse end-to-end.
- The structural ↔ nominal duality is operational: structural
  records can pass to nominal-typed parameters via row-polymorphism;
  nominal records pass to structural-typed parameters via TName
  unfolding through env lookup.

---

## What H2.3 reveals

- **RecordSchemeKind is the fourth SchemeKind variant.** SchemeKind's
  total_variants invariant (per H3) doesn't apply — records aren't
  "variants of an ADT". The ADT might be a degenerate single-variant
  ADT semantically, but its env presence is "record type", not
  "constructor". The split into RecordSchemeKind vs ConstructorScheme
  honors the user-visible distinction.

- **`{...}` is now FOUR things in expression position.** Block,
  bare record literal, nominal record literal (with prefix), record
  pattern (in match arm). Disambiguation is local and cheap, but
  the cognitive surface for Mentl users is real. SYNTAX.md should
  enumerate all four with examples.

- **Single-variant ADT vs nominal record converge to the same
  shape at runtime.** Both heap-allocate fields. The DIFFERENCE
  is named-vs-positional access. Mentl's audit (H5) gains another
  gradient candidate: "this single-variant tagged ADT could be a
  nominal record" — drop the tag word, gain `.field` access. Same
  WAT layout post-tag-removal.

- **TName chasing through env.** lookup_ty / resolve_field_offset
  may need to recursively unfold TName → RecordSchemeKind → fields.
  A small chase helper. Be aware of cycles (nominal record fields
  could reference each other). Cycle detection at registration
  time emits E_RecursiveType.

## Design synthesis (for approval)

**RecordSchemeKind(declared_fields)** added to SchemeKind ADT.

**TypeDefStmt with TLBrace body** parses as a single-variant
type whose body IS the record. register_one_variant detects this
case, binds env to RecordSchemeKind.

**NamedRecordExpr(type_name, fields)** — new Expr variant. Parser
detects type-prefixed `{...}` (capitalized ident + immediate brace).

**Inference** validates field set against RecordSchemeKind;
unifies values against declared field types; binds handle to
TName(type_name, []).

**Lower** routes NamedRecordExpr to LMakeRecord (no LIR change).

**Emit** unchanged.

**Field access** via resolve_field_offset extended for TName →
RecordSchemeKind lookup.
